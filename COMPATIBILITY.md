# Compatibility policy

## API

The HTTP API under `/v1` is additive within a minor version: fields may be
added to responses, new endpoints may appear, but existing fields, status
codes, and semantics do not change. Breaking changes require a new versioned
prefix (`/v2`) and a deprecation note in the release that introduces it. The
OpenAPI document generated by the running server is the authoritative
contract.

## Database

SQLite migrations in `migrations/` are ordered and immutable once released.
Newer Ravyn versions always upgrade older databases in place; downgrading is
not supported (restore the pre-upgrade backup instead — Ravyn's backup and
restore flow exists for exactly this). A populated-fixture migration test
guards legacy job preservation.

## External engines

Engine support is capability-based, not version-number-based:

- **yt-dlp** must support `--ignore-config`, `--dump-single-json`, `--print`,
  `--progress-template`, `--download-archive`, and `--ffmpeg-location`;
  installations missing any flag are rejected early with a structured error.
- **rqbit** must identify itself as `rqbit` at the API root and expose the
  torrent list/add/stats/add_peers/pause/start/delete/forget/update_only_files
  APIs; a missing API marks the engine incompatible instead of producing
  undefined behavior.

A numeric tested-version matrix is published once CI images with real engine
binaries exist; until then the capability contract above is the enforced
floor.

## Operating systems and toolchain

Releases are built and tested on Windows, Linux (x86_64), and macOS. The
minimum supported Rust version for building from source is pinned in CI
(currently stable plus a 1.85 floor) and only raised in a minor release.

## Configuration

Persisted settings are forward-compatible: unknown keys are rejected at write
time, and existing keys keep their meaning. Settings classified as
`restart_required` or `new_jobs_only` may become live in a later release, but
never the reverse.
