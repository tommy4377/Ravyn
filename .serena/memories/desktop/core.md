# Desktop shell (src-tauri, crate `ravyn-desktop`)
- `lib.rs` spawns the backend (`backend.rs`) in-process on an ephemeral loopback port and exposes native commands via `#[tauri::command]`. Decides setup-window vs main-window by re-reading `/v1/setup` state from the authenticated backend — never trust webview-local state.
- Current Tauri command surface (grep `lib.rs` for the authoritative list before assuming staleness — this list grows with features): `backend_info`, `setup_installation_info`, `apply_windows_integration`, `finish_setup_handoff`, `restart_application`, `main_window_ready`, `desktop_appearance`, `app_update_status`, `check_app_update`, `repair_application`, `cancel_app_update`, `install_app_update_now`, `open_native_path`, `reveal_native_path`, `browser_integration_status`, `repair_browser_integration`, `remove_browser_integration`, `take_browser_action`, `prompt_torrent_default_app`, `notify_native`, `open_compact_window`, `focus_main_window`. `open_native_path`/`reveal_native_path` provide Explorer-reveal/open. `open_compact_window`/`focus_main_window` back the always-on-top compact progress window (`tray.rs` builds the system tray icon that also targets the main window).
- **Single-instance is load-bearing, not cosmetic** (added 2026-07-18): `tauri_plugin_single_instance::init(...)` is registered as the FIRST plugin in the builder chain in `lib.rs::run()`. Without it, a magnet/`.torrent` OS-association launch (`torrent_association.rs` registers `"<exe>" "%1"` as the handler command) while Ravyn is already running spawns a fully redundant second process — second backend, second rqbit child, second window — racing the same SQLite DB. The plugin's callback re-parses argv via `parse_launch_action()` (covers both `--browser-action`/`--browser-source=` from Firefox relaunches and the raw magnet/`.torrent` association form) and forwards it into `BrowserActionState` + focuses the existing window, then the second process exits without ever calling `tauri::Builder::run`.
- `torrent_association.rs`: Windows per-user (`HKCU`) registration of Ravyn as a candidate handler for `.torrent`/`magnet:`. Split into `register(executable)` (pure registry writes, silent — called unconditionally from `integration::apply()`'s `register_installed_app` step so every fresh install is immediately torrent-capable) and `register_and_prompt()` (adds opening the Default Apps settings page — only reachable from the manual Settings/Torrents-view button). Never forces itself as the default; Windows owns that choice.
- Modules: `installation.rs`/`integration.rs` (installed-app registration, shortcuts, startup, Firefox native host registration), `setup_guard.rs`, `app_updates.rs` (updater phases: idle/checking/downloading/ready/installing/error, with rollback outcome reporting), `uninstall.rs` (also removes Firefox native host registration), `appearance.rs` (desktop wallpaper/accent introspection for the synthetic backdrop, see `docs/SYNTHETIC_BACKDROP.md`), `shell_paths.rs`, `browser_integration.rs` (Firefox native host manifest management, browser action queue, command-line parsing), `native_messaging.rs` (native messaging protocol listener, `BackendDescriptorGuard` to publish the backend port).
- `browser_integration.rs` provides:
  - `register()` / `unregister()` / `repair_for_current_executable()` for the per-user Firefox native-messaging host manifest in `HKCU:\Software\Mozilla\NativeMessagingHosts\com.ravyn.download_manager`.
  - `BrowserIntegrationStatus` struct reported to the frontend.
  - `BrowserActionState` (shared state for pending actions from the extension) and `take_browser_action` Tauri command.
  - `try_handle_command_line()` / `parse_browser_action()` for the `--browser-action` CLI arg.
- `native_messaging.rs` provides:
  - Native messaging protocol listener that proxies extension requests to the backend API.
  - `BackendDescriptorGuard` that writes/cleans up the native host manifest with the dynamic backend port.
  - `try_handle_command_line()` for the `--native-messaging` CLI arg (Firefox native messaging pipe mode).
- Install directory equals the data directory, so updater rollback must remain binaries-only — never roll back user data.
- A debug-only `tauri-plugin-mcp-bridge` dependency enables `tauri-mcp` tooling; do not treat it as present in release builds.
- Crate type is `staticlib`/`cdylib`/`rlib`; depends on root `ravyn` crate via path dependency.
- App icon (`src-tauri/icons/`) is embedded into the exe as a Windows resource by `build.rs` at compile time only. Force icon rebuild by touching `src-tauri/build.rs`. Regenerate icons with `npx @tauri-apps/cli icon <source>.svg -o src-tauri/icons`; delete iOS/Android/`.icns`/Appx artifacts (only the `.ico`/`.png` pair is used).
- Windows distribution (since 2026-07-17): a single self-installing `Ravyn.exe` — no NSIS or MSI bundles (`bundle.active: false`). The in-app setup performs installation into `%LOCALAPPDATA%\Ravyn`; the updater helper replaces the installed binary in place and refreshes `DisplayVersion`; `webview_runtime.rs` guards startup when the WebView2 runtime is missing. Firefox native host registration is runtime-only (`--register-firefox-native-host`).
