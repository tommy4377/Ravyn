# Desktop shell (src-tauri, crate `ravyn-desktop`)
- `lib.rs` spawns the backend (`backend.rs`) in-process on an ephemeral loopback port and exposes native commands via `#[tauri::command]`. Decides setup-window vs main-window by re-reading `/v1/setup` state from the authenticated backend — never trust webview-local state.
- Current Tauri command surface (grep `lib.rs` for the authoritative list before assuming staleness — this list grows with features): `backend_info`, `setup_installation_info`, `apply_windows_integration`, `finish_setup_handoff`, `restart_application`, `main_window_ready`, `desktop_appearance`, `app_update_status`, `check_app_update`, `repair_application`, `cancel_app_update`, `install_app_update_now`, `open_native_path`, `reveal_native_path`. `open_native_path`/`reveal_native_path` provide Explorer-reveal/open — do NOT assume this is unimplemented (an older note claimed "exactly 5 setup-only commands, no reveal-in-Explorer"; that was true only as of the 2026-07-13 setup milestone and is now false).
- Modules: `installation.rs`/`integration.rs` (installed-app registration, shortcuts, startup — gated on recorded consent read from the backend), `setup_guard.rs`, `app_updates.rs` (updater phases: idle/checking/downloading/ready/installing/error, with rollback outcome reporting), `uninstall.rs`, `appearance.rs` (desktop wallpaper/accent introspection for the synthetic backdrop, see `docs/SYNTHETIC_BACKDROP.md`), `shell_paths.rs`.
- Install directory equals the data directory (see `mem:conventions`), so updater rollback must remain binaries-only — never roll back user data.
- A debug-only `tauri-plugin-mcp-bridge` dependency enables `tauri-mcp` tooling; do not treat it as present in release builds.
- Crate type is `staticlib`/`cdylib`/`rlib`; depends on root `ravyn` crate via path dependency, so backend and desktop shell always build against the same in-tree backend version.
