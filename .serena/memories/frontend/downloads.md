# Downloads vertical slice (first complete main-app screen)

- `lib/downloads/jobPresentation.ts::permittedActions(status, kind)` is the single source of truth for which job actions are shown/enabled. It mirrors the exact transition guards in `src/core/lifecycle.rs` (pause/resume/retry/cancel) — do not infer permitted actions ad hoc elsewhere. Notable non-obvious rules: pause only allowed from `probing`/`downloading` (+`seeding` for torrents only); resume allowed from both `paused` AND `failed`; retry allowed from `failed`/`cancelled`/`partial`; cancel allowed for any non-terminal status (no backend guard, UI-side policy).
- `DELETE /v1/jobs/{id}` (and bulk `"delete"` action) does NOT delete downloaded files for http/media jobs — only cancels + removes the DB row. UI label is always "Remove from list", never bare "Delete" (see `lib/downloads/jobActions.ts`).
- Reveal/open-in-Explorer IS implemented via the desktop shell's `open_native_path`/`reveal_native_path` Tauri commands (see `mem:desktop/core`) — the earlier note that no such command existed only held as of the 2026-07-13 setup milestone; confirm current wiring in `lib/native/tauri.ts` before assuming either presence or absence of a UI entry point for it.
- `POST /v1/jobs/actions` bulk endpoint treats an **empty `ids` array as "apply to all jobs"**. Frontend never sends empty arrays — command-bar bulk buttons are `disabled` whenever selection is empty, which structurally prevents this footgun.
- Progress SSE events (`type: "progress"`) are coalesced to one shared ~10Hz flush timer in `jobsStore` (`stores/jobs.svelte.ts`), not per-entity timers — `job_status` events apply immediately (low frequency, cheap).
- Downloads view tabs (Active/Queued/Completed/Failed/All) are pure client-side filters over one normalized, paginated collection (`jobsStore.jobsFor(view)`), not separate server queries — backend only supports a single-value `status` filter, not multi-status OR.
- `VirtualList.svelte` is a generic (`generics="T"`) windowed list; keyboard nav lives on ITS listbox element (`role="listbox"`, `tabindex="0"`, real `aria-activedescendant`), not on an outer wrapper — putting `onkeydown` on a non-interactive wrapper div is an a11y-linter error and was a real bug caught by `npm run check`.
- `Menu.svelte`/`ContextMenu.svelte`/`MenuButton.svelte` are hand-built (no bits-ui dependency): `Menu` takes explicit `x`/`y` fixed-position coords, not an anchor ref.
- `lib/downloads/jobActions.ts::buildJobMenuItems` + `jobPresentation.ts::permittedActions` are NOT downloads-only despite the folder — `lib/torrents/TorrentsView.svelte` reuses them verbatim (as of 2026-07-18) for row-level pause/resume/retry/cancel/remove parity. Any new job-kind-specific view should follow the same pattern rather than reimplementing action gating from engine-native state strings.
- **`jobsStore` (stores/jobs.svelte.ts) is a live, global store fed by AppShell's SSE subscription regardless of which page is active** — `AppShell.svelte`'s `$effect` calls `jobsStore.applyEvent(event)` for every `job_status`/`progress` SSE event unconditionally, not just while `DownloadsView` is mounted. As of 2026-07-18, `TorrentsView.svelte` and `MediaView.svelte` both exploit this: after their own `listJobs({kind:...})` fetch, they call `jobsStore.upsert(job)` for each result (seeds `byId` without disturbing the store's own pagination/`order`/filter state used by `DownloadsView`), then derive a live-merged display object per row — `jobsStore.byId.get(job.id)` for status, `jobsStore.liveProgress.get(job.id)` for `downloadedBytes`/`totalBytes`/`bytesPerSecond` — falling back to their own locally-fetched snapshot when no live entry exists yet. This is the correct pattern for ANY future job-kind view: don't maintain a fully separate live-status mechanism, seed `jobsStore` and read it back. Torrent-only fields with no SSE channel (upload speed, peers, seeders/leechers, ratio — backend only reports those into internal Prometheus metrics via `torrent_telemetry`, not `Event::Progress`) still need periodic polling; both views re-`load()` every 5s while mounted for that reason, matching the backend's own `adapters::torrent::monitor_managed` 5s reconciliation cadence.
- `stores/connection.svelte.ts::connect()` closes any previously-open `RavynEventClient` (`this.events?.close()`) before creating a new one (fixed 2026-07-18) — without this, a retry after `mainWindowReady()` rejected post-connect (or any repeated `connect()` call) leaked the prior live `EventSource`, since nothing else held a reference to it once `this.events` was reassigned.
- `components/VirtualList.svelte`'s scroll-to-index `$effect` reads `scrollTop` via `untrack()` (fixed 2026-07-18) — it's both an input and an output of that same effect (computes `next` from current `scrollTop`, then writes `scrollTop = next`), and without `untrack` the write counts as a change to one of the effect's own dependencies, scheduling a redundant second (no-op) run on every scroll-to-index call (e.g. every arrow-key navigation in a large list).
