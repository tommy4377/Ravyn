# Frontend map
- Real Svelte 5 (runes) + Vite + TypeScript app under `frontend/`, serving both the setup flow and the main application. Not a stub — full feature surface exists.
- `lib/api/` — `transport.ts` (fetch/timeout/abort), `client.ts` (typed per-endpoint methods), `types.ts` (mirrors backend contracts, keep in sync with `/openapi.json`), `errors.ts` (stable backend error codes; network failures map to `NETWORK_UNAVAILABLE`), `events.svelte.ts` (SSE).
- `lib/native/tauri.ts` is the only place `@tauri-apps/api` `invoke()` is called from app code. Current surface includes: `backendInfo`, `appUpdateStatus`, `checkAppUpdate`, `installAppUpdateNow`, `cancelAppUpdate`, `desktopAppearance`, `setupInstallationInfo`, `applyWindowsIntegration`, `finishSetupHandoff`, `restartApplication`, `mainWindowReady`, `repairApplication`, `openNativePath`, `revealNativePath`, `pickFolder`, `browserIntegrationStatus`, `repairBrowserIntegration`, `removeBrowserIntegration`, `takeBrowserAction`.
- `lib/stores/*.svelte.ts` — rune-based global stores: connection, jobs, navigation, notifications, selection. Colocated `*.test.ts` run under vitest.
  - Navigation store: supports `requestAdd(kind, sourceURL)` with `pendingAddSource` for browser action source URLs.
- `lib/setup/` — setup flow (`SetupApp.svelte`, `controller.svelte.ts`, `stages/*Stage.svelte`, `installationPolicy.ts`, `componentStates.ts`).
- `lib/shell/` — main app chrome: `AppShell.svelte` (includes 750ms browser action polling via `takeBrowserAction`), `NavigationView.svelte`, `CommandBar.svelte`, `StatusBar.svelte`, notification host/drawer, `ConnectionBoot.svelte`.
- Feature areas, each with a `*View.svelte` root and presentation-logic `.ts` files kept separate from markup for testability: `downloads/` (`mem:frontend/downloads`), `library/`, `media/`, `torrents/`, `automation/`, `basket/`, `settings/` (category-based with `SettingsCategoryHeader.svelte` + per-category `*Settings.svelte` — categories include `browser` for `BrowserIntegrationSettings`), `diagnostics/`, `appearance/`.
- `lib/settings/BrowserIntegrationSettings.svelte` displays the current Firefox native host registration status with repair/remove buttons.
- `lib/components/` — hand-built shared UI primitives (Fluent Design 2 structure, "Signal" visual theme), no external component-library dependency.
- Design tokens, the always-on system-accent override, and the canonical logo asset: `mem:frontend/design_system`.
- Pattern for every feature: presentation logic (pure, unit-testable `.ts`) is separated from the `.svelte` view that wires it to stores/API — follow this split when adding features rather than putting logic inline in components.
