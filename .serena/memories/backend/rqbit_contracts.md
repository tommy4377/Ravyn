# rqbit contracts
- HTTP compatibility is capability-based; do not infer support from a version string.
- Every rqbit success JSON body is streamed through a 4 MiB hard limit; error bodies are capped at 8 KiB.
- DHT stats expose a Ravyn-owned contract for `id`, `outstanding_requests`, `routing_table_size`, and `routing_table_size_v6`.
- DHT table normalization requires top-level `v4` and `v6`; reject more than 16,384 JSON nodes, depth over 32, and strings over 4 KiB.
- A 404 from DHT endpoints means the capability is disabled or unsupported and maps to `Unavailable`, not a missing Ravyn resource.
- Torrent payload byte rate control remains blocked until rqbit exposes a verified runtime API governing actual transfer bytes.
- **rqbit has no session persistence** (no `--session-persistence` flag passed by `services/rqbit_process.rs`): its in-memory torrent-id counter restarts from zero on every process launch. Any `torrent_jobs` row surviving from a previous run is therefore guaranteed to collide with a freshly issued id on the `torrent_jobs.torrent_id` unique index (sqlite code 2067). Fixed 2026-07-18: `TorrentAdapter::new` calls `Repository::clear_torrent_records()` once at construction (1:1 with a fresh rqbit process spawn per app launch), and `upsert_torrent_record` additionally retries once after deleting the conflicting row if a unique-violation slips through anyway (e.g. a mid-session rqbit child respawn via `RqbitProcessManager`, which doesn't reconstruct `TorrentAdapter`). Don't reintroduce a plain `INSERT ... ON CONFLICT(job_id)` without this safety net.
