- **`services::filename::sanitize` truncation ordering** (fixed 2026-07-18): truncating to the 240-char limit BEFORE re-trimming can reintroduce a trailing `.`/` ` that a prior trim step removed (Windows silently drops these from paths it actually creates, so the on-disk name can diverge from what's recorded). The function now trims again after truncating. `core::lifecycle::validate_filename` also didn't reject `".."`/`"."` (both parse to exactly one `Path` component, satisfying the old `components().count() != 1` check despite the error message's claim) — now additionally requires that component be `Component::Normal`. See tests `services::filename::tests::truncation_does_not_reintroduce_a_trailing_dot_or_space` and `core::lifecycle::validate_filename_tests::*`.
